## Архитектура

### Общая схема

SPA на Vue 3 общается с Laravel-бэкендом через REST API с Sanctum аутентификацией. 
Тяжёлая операция парсинга вынесена в очередь (database-драйвер). 
Прогресс транслируется на фронт через Server-Sent Events (SSE).

### Бэкенд (Laravel)

Представляет собой стандартное Laravel приложение. Парсинг вынесен в отдельный сервисный слой (`app/Services/YandexMaps/`), 
тяжёлая работа - в очередь через Job, общение с фронтом - через SSE.

### Фронтенд (Vue 3 + TypeScript)

Представляет собой обычное Vue приложение. Код расположен в `resources`, точкой входа является
`app.blade.php`. Приложение придерживается FSD.

Используемые технологии: 
- vue-router
- tanstack-query (работа с запросами, клиентский кэш, инвалидация)
- daisyui (ui библиотека поверх tailwindcss)
- ofetch (легковесный http клиент)

## Парсинг

В приложении используется следующая стратегия парсинга:
- Обрабатывается переданные пользователем url, оттуда извлекается ogrId (если короткая ссылка, то делается дополнительный 
запрос без перехода по редиректу для получения полной ссылки)
- Делатеся HTTP Get запрос на получение страницы организации в картах. Оттуда сохраняются куки и при помощи простого
парсера извлекается основная информация (название, рейтинг, количество оценок/отзывов), csrf токен и sessionId (он же используется как reqId).
Основная информация записывается в БД, отправляется событие `info_ready` (его подхватывает фронт и запрашивает информацию про организацию, чтобы пользователь
хотя бы ее видел, пока ждет получение отзывов)
- При помощи извлеченных ранее куки, csrf и reqId делаются параллельные запросы к апи яндекса, откуда получается __полная__ информация
про отзывы. Запросов делается несколько, пока все отзывы не будут получены (12 страниц, по 50 на каждой, дальше апи отдает ошибки). 
Эта информация записывается в БД, отправляется событие `reviews_ready`. 

Также у пользователя есть возможность запросить `refresh` данных об отзывах организации (на это установлен лимит, который редактируется в конфиге приложения).

Тут появляется вопрос - зачем сохранять отзывы в своей БД, если можно просто направлять их напрямую к апи Яндекса? На это я 
нашел пару причин: 
- Расширяемость. Если со временем появятся новые, то мы их просто добавляем в базу, старые не пропадают
из-за лимита апи карт.
- Удобство. Если отзывы нужны будут для проведения какой-то бизнес аналитики, то мы можем их просто 
взять из базы.

Преимуществом данной стратегии парсинга является скорость, производительность и малая зависимость 
от изменений в фронтенде яндекс карт. Вместо поднятия headless браузера или анализа html мы просто
делаем http запросы, из которых можно извлечь максимум информации. 

Помимо этого, во время разработки не были замечены какие-либо ограничения (антиспам, защита от ботов) на
запросы к апи для получения отзывов. На всякий случай в конфиг добавлена возможность установить задержку
между запросами.
