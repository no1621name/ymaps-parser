## Архитектура

### Общая схема

SPA на Vue 3 общается с Laravel-бэкендом через REST API с Sanctum аутентификацией. 
Тяжёлая операция парсинга вынесена в очередь (database-драйвер). 
Прогресс транслируется на фронт через Server-Sent Events (SSE).

### Бэкенд (Laravel)

Представляет собой стандартное Laravel приложение. Парсинг вынесен в отдельный сервисный слой (`app/Services/YandexMaps/`), 
тяжёлая работа - в очередь через Job, общение с фронтом - через SSE.

### Фронтенд (Vue 3 + TypeScript)

Представляет собой обычное Vue приложение. Код расположен в `resources`, точкой входа является
`app.blade.php`. Приложение придерживается FSD.

Используемые технологии: 
- vue-router
- tanstack-query (работа с запросами, клиентский кэш, инвалидация)
- daisyui (ui библиотека поверх tailwindcss)
- ofetch (легковесный http клиент)

## Парсинг

В приложении используется двухуровневая стратегия парсинга с автоматическим fallback.

### Основной путь — API-парсер (`ApiReviewParser`)

- Обрабатывается переданный пользователем url, оттуда извлекается ogrId (если короткая ссылка, то делается дополнительный
запрос без перехода по редиректу для получения полной ссылки)
- Делается HTTP GET запрос на получение страницы организации в картах. Оттуда сохраняются куки и при помощи простого
парсера извлекается основная информация (название, рейтинг, количество оценок/отзывов), csrf токен и sessionId (он же используется как reqId).
Основная информация записывается в БД, отправляется событие `info_ready` (его подхватывает фронт и запрашивает информацию про организацию, чтобы пользователь
хотя бы ее видел, пока ждет получения отзывов)
- При помощи извлеченных ранее куки, csrf и reqId делаются параллельные запросы к API Яндекса (`Http::pool`, concurrency = 3),
откуда получается __полная__ информация про отзывы. Запросов делается несколько, пока все отзывы не будут получены (12 страниц,
по 50 на каждой, дальше API отдаёт ошибки). Эта информация записывается в БД, отправляется событие `reviews_ready`.

### Деградация при rate-limit (429)

При получении ответа 429 от API Яндекса парсер автоматически переходит с параллельных запросов на последовательные
с рандомной задержкой между ними (от 1 до 3 секунд, настраивается в `config/yandex_maps.php` через `min_delay_ms` / `max_delay_ms`).
При повторном 429 в последовательном режиме — одна попытка повтора с задержкой `max_delay_ms`.
Переключение в sequential mode логируется.

### Fallback — headless-браузер (`HeadlessReviewParser`)

Если API-парсер падает с любой ошибкой (сломанная подпись, капча, пустой ответ при ненулевом счётчике отзывов),
`ParserOrchestrator` автоматически переключается на headless-парсер. Он использует `spatie/browsershot` (Puppeteer/Chromium)
для загрузки страницы организации, скроллит контейнер отзывов для подгрузки ленивых элементов (до 50 попыток),
затем парсит DOM через Symfony `DomCrawler`. Оба парсера реализуют интерфейс `ReviewParser`, что позволяет легко
добавлять новые стратегии сбора данных.

### Canary-проверка

Команда `php artisan yandex-maps:canary` выполняет ежедневную (зарегистрирована в `routes/console.php` как `->daily()`)
проверку работоспособности API Яндекса. Проходит полный pipeline (`fetchOrgPage` → `extractSessionData` → `fetchReviews`),
не записывая данные в БД — read-only проверка. При успехе логирует метаданные организации, при ошибке — пишет
в `Log::error` с трейсом.

### Refresh и хранение отзывов

У пользователя есть возможность запросить `refresh` данных об отзывах организации (на это установлен лимит,
который редактируется в конфиге приложения).

Зачем сохранять отзывы в своей БД, если можно направлять их напрямую к API Яндекса?
- **Расширяемость.** Если со временем появятся новые, то мы их просто добавляем в базу, старые не пропадают из-за лимита API карт.
- **Удобство.** Если отзывы нужны будут для проведения какой-то бизнес-аналитики, то мы можем их просто взять из базы.


## Тестирование

Сетевой путь парсинга покрыт тестами на фикстурах — реальных запросов к API Яндекса нет, весь HTTP-трафик подменяется через `Http::fake()`.

### Фикстуры

Файлы в `tests/Fixtures/YandexMaps/`:
- `org-page-success.html` — HTML-страница организации
- `reviews-page-*.json` — JSON-ответы API с отзывами (разные сценарии пагинации)
- `csrf-only-response.json` — ответ с истёкшей сессией
- `empty-response.json` — пустой ответ
- `rate-limit-response.txt` — ответ 429

### Модульные тесты (`tests/Unit/ApiClientTest.php`)

10 тестов, покрывающих:
- Успешное получение страницы организации и отзывов
- Обработку 429 (rate-limit) — fallback на sequential mode при 429 на первой странице и на пул-запросах
- CSRF-expired сессию
- Пустые ответы и ранний выход при неполной странице
- Двухстраничную и одностраничную пагинацию

### Интеграционные тесты (`tests/Feature/ParserOrchestratorTest.php`)

3 теста с `RefreshDatabase`:
- Полный pipeline: HTML + 2 страницы отзывов → проверка созданных записей, статусов и SSE-событий
- Ошибка API → статус `Failed` + событие `failed`
- CSRF-expired → каскадная ошибка → `Failed`

### Headless-парсер (`tests/Unit/HeadlessReviewParserTest.php`)

Тесты на парсинг HTML от headless-браузера: извлечение метаданных организации (название, рейтинг, количество отзывов)
и данных отдельных отзывов (автор, текст, дата, лайки) из DOM-фрагментов.

## Что можно улучшить

- Перенести очередь с database провайдера на Redis. На данный момент это не было сделано из-за
небольшой нагрузки на очередь и малой мощности VPS, на которой проект поднят
- Возможность полного повторного парсинга сведений об организации
- Настроить CI/CD
- Поработать над SSE
- Оптимизация headless-парсинга (сейчас скроллит до 50 раз с интервалом 2 секунды)
- Более умная стратегия ретраев при 429 (например, exponential backoff вместо единичного повтора)
